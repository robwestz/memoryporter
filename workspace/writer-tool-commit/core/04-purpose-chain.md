# Purpose chain - writer-tool-commit

## User outcome
A content professional produces a piece their editor cannot distinguish from a human-written one, in under 5 minutes of work, for the niche they already operate in.

## Intermediate effects
- User picks a skill-template matching their genre (e.g., "Bacowr guest post")
- User fills a short brief (topic, angle, source URL or doc)
- Skill-template generates structured output with source-grounded citations
- User reviews and the content is at editor-ready baseline, not "edit into shape"
- User ships and gets paid for it (or publishes and gets traction)

## Step-to-step dependencies
- Brief quality bounds output quality — UX must make a good brief easy
- Skill-template quality bounds structural correctness — skill-forge is the upstream factory
- Source-grounding layer bounds trust — without real citations, trust collapses before any UX win registers

## Global success conditions
- At least one real paying user uses the tool weekly after 90 days
- At least one real editor signs off on a piece without knowing it was AI-assisted
- Revenue > cost within 180 days (not profitability, just self-sustaining)

## Local optimizations to avoid
- Adding more skill-templates before the first one passes editor-blind-test — catalog breadth with no depth is fake
- Polishing UI before generation quality is proven — pretty wrappers on bad output waste runway
- Building a generic "any-niche" interface early — removes the scaffolding advantage the product is built on
- Building payment/billing before a free user has returned voluntarily twice
