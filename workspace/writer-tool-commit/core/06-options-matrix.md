# Options matrix - writer-tool-commit

## Standard options
- Option A: **Standalone product** — writer-tool as its own SaaS (Next.js + Supabase, new domain, separate from Bacowr). Ship first skill-template end-to-end in 8 weeks.
- Option B: **Bacowr generation engine** — writer-tool becomes the generation layer INSIDE Bacowr. No new SaaS, no new domain, no new auth. Bacowr users get skill-template output in the existing workflow.
- Option C: **Portable-kit skill package** — writer-tool collapses to a skill-forge-authored meta-skill. No product, no customers, no launch. Skill that Robin (or consulting clients) use directly.
- Option D: **Kill-and-pivot** — abandon writer-tool, focus remaining capacity on whichever of RepoBrain/Bacowr is closer to revenue. Accept sunk cost on night-project A.

## Comparison matrix
| Option | Preserves | Changes | Evidence needed | Main risk |
|---|---|---|---|---|
| A | blueprint standalone vision; reuses night-C launch assets | adds fourth product surface; duplicates Bacowr auth/billing | 8-week ship proof; editor-blind-test pass | Bacowr cannibalization; attention split |
| B | Bacowr existing user path and infra; kills the "four-products" problem | subordinates writer-tool branding; harder to sell broadly | Bacowr user willingness to pay more for skill-template output | Niches outside Bacowr go untapped |
| C | Robin capacity; portable-kit compounding | any product/revenue ambition | consulting demand; whether skills alone generate income | Regret if market window closes |
| D | remaining capacity for closer-to-ship products | all writer-tool work to date | clear evidence RepoBrain or Bacowr is closer to paying users | Abandoning sunk work that is genuinely good |

## Recommendation posture
- Preferred option and why: **B (Bacowr generation engine)**. It resolves the four-products problem by making writer-tool power Bacowr rather than compete with it. Preserves the skill-template IP. Uses Bacowr's real users as the editor-blind-test pool. Cuts ship-surface from "new SaaS" to "new feature."
- Why the other options remain valid: A if evidence emerges that Bacowr users are the wrong first niche. C if Robin's capacity is genuinely smaller than assumed. D if the first template fails editor-blind-test and the signal is "market does not want this."

## Anti-simplification review

1. What existing safeguard would disappear if this option succeeded locally?
   - A: Bacowr focus — pulling Robin's attention creates a quieter failure where Bacowr stalls while writer-tool gets the new-project glow.
   - B: Writer-tool's independent brand — becoming a "Bacowr feature" makes pivoting to non-Bacowr niches require a re-launch.
   - C: The editor-blind-test discipline — without paying users, quality claims drift back to Robin's own read.
   - D: The skill-template catalog IP — abandoning writer-tool loses the prompt-architecture insight that drove it.
2. Which output looks better in isolation but weakens the global purpose chain?
   - A: "new standalone SaaS" looks like a real product but adds a fourth surface with no ship closer.
   - B: "Bacowr got better" is quieter than a launch; in the global chain it is stronger because it compounds existing revenue.
   - C: "new skill-package" feels productive but explicitly gives up on the product ambition the blueprint was for.
   - D: "focus" looks disciplined but is the same move that tabled writer-tool on 2026-04-10 — repetition masquerading as decision.
3. Which assumption is being smuggled in as if it were proven?
   - A: That content pros outside Bacowr's niche will find writer-tool as a standalone product.
   - B: That Bacowr users actually want generation-quality enough to pay more (vs better reporting, more links, cheaper price).
   - C: That consulting or skill-packaging income is comparable to SaaS income — unproven.
   - D: That RepoBrain or Bacowr alone can clear the revenue bar without writer-tool's IP.
4. What would a false-positive "improvement" look like here?
   - A: 500 launch-day signups that do not return — vanity metric, no validation.
   - B: Bacowr customers say "nice" in support chat — enthusiasm without willingness to pay more.
   - C: Robin feels productive authoring skills nobody pays for — build-vs-ship pattern wins again.
   - D: Relief at "simplifying" that turns into regret 3 months later — repeat of 2026-04-10 tabling.
