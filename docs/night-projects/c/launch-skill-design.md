## 3. Core Mechanism — The Narrative Spine

**The key insight from research:** Stripe, Linear, and Vercel all launched with zero paid acquisition because their story was consistent across every touchpoint. Every framework (YC deck, landing page, email drip) opens on the problem, drips the reveal, leads social proof before the ask. Inconsistency across assets is the most common failure mode in launch packages generated by AI.

**The solution:** Derive a Narrative Spine before generating any deliverable. All 6 templates ingest the spine as shared context. If the spine is correct, narrative consistency is structural — not a review step.

### Spine Structure

```yaml
# narrative-spine.yaml (internal — not saved, used as generation context)

hook_statement: >
  [ICP] spend [time/money] on [problem] — and it doesn't have to be that way.

problem_in_customer_language: >
  [The pain as the customer would say it — not how the founder describes it]

why_now: >
  [The tailwind — tech shift, regulatory change, market timing — that makes this possible today]

solution_promise: >
  [Outcome] for [ICP] without [obstacle]
  Formula: "[Verb] [Outcome] [Time Frame or Qualifier]"

unique_mechanism: >
  [The one thing the product does that competitors cannot honestly claim]

proof_signals:
  - [Traction datum 1 — users, revenue, growth rate, notable customer]
  - [Traction datum 2]
  - [Testimonial quote — specific outcome + named person]

canonical_cta: >
  [Outcome verb] + [what they receive] + [optional qualifier]
  Example: "Get your first report free" not "Sign up"

voice_adjectives:
  - [e.g., direct]
  - [e.g., technical]
  - [e.g., no-fluff]

icp_one_liner: >
  [Role] at [company type/size] who [specific behavior or struggle]
```

**Generation order:** The spine is derived first. Landing page copy is written second (most detailed copywriting — establishes canonical phrasing). Each subsequent deliverable adapts canonical phrasing to its format constraints. This means copy flows forward, not backward.

**Canonical language flows:**
```
Spine → Landing Page → Pitch Deck → Email Sequence → Product Hunt → Social Calendar → Demo Script
```

## 4. Input Specification

### Required inputs (blueprint must contain all of these)

| Field | Type | Notes |
|-------|------|-------|
| `product_name` | string | The product's name |
| `icp` | string | Role, industry, company size — be specific |
| `problem` | string (1–3 sentences) | The pain — in customer language |
| `solution` | string (1–3 sentences) | What the product does |
| `unique_mechanism` | string | The differentiator — what competitors can't honestly say |
| `pricing` | string | Model + entry price (e.g., "freemium, $29/mo Pro") |
| `tone` | string[] | 2–4 voice adjectives (e.g., "direct, warm, no-jargon") |
| `launch_date` | date (YYYY-MM-DD) | T-0 — used to anchor the social calendar |

### Optional inputs (output quality improves with each)

| Field | Type | Used by |
|-------|------|---------|
| `tagline` | string | Product Hunt (starting point), Landing page |
| `traction` | string | Proof signals — users, revenue, MoM growth |
| `testimonials` | string[] | Social proof sections in LP, emails, social |
| `competitor_names` | string[] | Pitch deck competition slide |
| `team_bios` | string[] | Pitch deck team slide |
| `raise_amount` | string | Pitch deck ask slide |
| `why_now` | string | If provided, use verbatim; otherwise derive from category context |

### Validation rules (run before generation)

- If `unique_mechanism` is absent or generic ("it's faster") → halt and ask. This field propagates everywhere; a weak mechanism produces uniformly weak copy.
- If `launch_date` is missing → default social calendar to relative dates (Day 1, Day 2...) but note the limitation.
- If `traction` is empty → flag in QA report: proof sections will be placeholder-only.

## 5. Workflow

```
INTAKE → SPINE → GENERATE (×6) → QA → REPORT
```

### Phase 0: INTAKE

Parse the product + blueprint. Build the internal context object. Validate required fields. If any required field is missing, ask before proceeding — never invent product facts.

Output: validated input context object. Not saved.

### Phase 1: SPINE

Derive the Narrative Spine from the input context. Apply formulas:

- `solution_promise`: `[Outcome] for [ICP] without [Obstacle]`
- `hook_statement`: `[ICP] spend [X] on [problem] — and it doesn't have to be that way.`
- `canonical_cta`: Outcome verb + what they receive + optional qualifier
- `why_now`: derive from category if not provided (AI shift, remote work, regulation, etc.)

**Gate:** Before moving to GENERATE, state the spine explicitly so the user can correct it. The spine is the single point of failure — a wrong spine produces 6 wrong deliverables.

Output: stated Narrative Spine (visible to user).

### Phase 2: GENERATE

Produce all 6 deliverables in canonical order. For each:
1. Copy the matching template from `templates/`
2. Fill all `[VARIABLE]` zones from the spine + input context
3. Preserve all `[FIXED]` zones exactly

**Generation order and why:**

| Order | Deliverable | Why this position |
|-------|------------|-------------------|
| 1 | Landing page copy | Establishes all canonical language — longest form, most decisions |
| 2 | Pitch deck outline | Adapts LP language to investor context; adds market/financials |
| 3 | Email sequence | Adapts LP language to email conversation; reveals in sequence |
| 4 | Product Hunt listing | Most constrained — distills canonical language to tight limits |
| 5 | Social calendar | Derives content pillars from LP headlines + email subjects |
| 6 | Demo script | Uses canonical language; builds toward the specific wow moment |

### Phase 3: QA

Run Layer 1 (structural) then Layer 2 (narrative). Both always run. A Layer 1 pass does not mean assets are good.

### Phase 4: REPORT

Fill `templates/launch-package-report.md`. State verdict. List 3 priority fixes ordered by impact.

## 6. Deliverable Specifications

### 6.1 Landing Page Copy

**Template:** `templates/landing-page.md`

**Structure (FIXED zones = section headings; VARIABLE = content):**

```
HERO
  [VARIABLE] Headline — Formula: [Verb] [Outcome] [Qualifier]
  [VARIABLE] Subheadline — 1 sentence, specific, adds a number or outcome
  [FIXED]    Social proof hook placement — immediately under subheadline
  [VARIABLE] Social proof hook — "Trusted by X teams" or star rating + count
  [FIXED]    CTA button placement
  [VARIABLE] CTA button text — outcome language, action verb
  [VARIABLE] Hero visual descriptor — product in use (not logo, not abstract)

PROBLEM SECTION
  [VARIABLE] Problem statement — 2–3 sentences, customer language, mirror their words

BENEFIT SECTION 1
  [FIXED]    3-column grid format: Icon | Benefit Name | 1-line statement
  [VARIABLE] 3 benefit rows — lead with outcome, not feature name

BENEFIT SECTION 2
  [VARIABLE] 3 more benefit rows (different cluster of value)

BENEFIT SECTION 3
  [VARIABLE] 3 more benefit rows (third cluster — often: trust, scale, integration)

HOW IT WORKS
  [FIXED]    3-step numbered sequence format
  [VARIABLE] Steps — path from signup to value, must feel short

SOCIAL PROOF
  [VARIABLE] Testimonial 1 — specific outcome + full name + title + company
  [VARIABLE] Testimonial 2
  [VARIABLE] Testimonial 3
  [VARIABLE] Logo strip — list customer names/logos
  [VARIABLE] Number proof — "X customers", "Y% faster", "$Z saved/month"

FAQ
  [FIXED]    3–5 Q&A pairs
  [VARIABLE] Q1 — pricing objection
  [VARIABLE] Q2 — security/trust objection
  [VARIABLE] Q3 — "is this right for me" objection
  [VARIABLE] Q4 — optional (integration/compatibility)
  [VARIABLE] Q5 — optional (support/onboarding)

FINAL CTA
  [FIXED]    Repeat hero CTA format
  [VARIABLE] Risk reducer — "No credit card required" / "Cancel anytime" / "Free trial"
  [VARIABLE] Urgency signal (if available)
```

**Hard constraints:**
- Headline: ≤ 12 words
- Subheadline: ≤ 25 words
- Social proof hook: ≤ 8 words
- CTA button text: 2–5 words
- Each benefit statement: ≤ 12 words
- FAQ answers: 2–4 sentences each

### 6.2 Pitch Deck Outline

**Template:** `templates/pitch-deck.md`

**12-slide structure (derived from YC format, adapted for product launch context):**

| Slide | Title | VARIABLE content | Speaker note |
|-------|-------|-----------------|-------------|
| 1 | Title | Product name + punchline (= solution_promise from spine) | 1 sentence — why this company exists |
| 2 | Problem | Who hurts + quantified cost of problem | Key stat — makes audience feel the pain |
| 3 | Solution | Product promise in 1 sentence + demo visual descriptor | Demo is the pitch — describe it |
| 4 | Demo | 3-step flow: before → action → after | Make the "after" feel inevitable |
| 5 | Market | TAM/SAM/SOM with named source | SAM is the number that matters to investors |
| 6 | Traction | Revenue / users / MoM growth rate + chart shape | Show slope, not just number |
| 7 | Business Model | Who pays, how much, gross margin estimate | Unit economics matter more than ARR |
| 8 | Competition | 2-axis quadrant — X = [dim1], Y = [dim2], you top-right | Name the axes to define the game |
| 9 | Team | Photo + one past win per person + why us for this | Past wins predict future wins |
| 10 | Financials | 18-month projection: burn, revenue, milestones | Credible ≠ precise — show the logic |
| 11 | Ask | Raising $X for Y milestones (2–3 specific) | Milestones must be fundable events |
| 12 | Vision + Contact | 10-word domination statement + contact + QR | End on ambition, not logistics |

**Hard constraints:**
- Each slide: title + 3–5 bullet points + 1 speaker note (≤ 3 sentences)
- Competition axes: must be derived from the unique_mechanism — not generic "price vs. features"
- Traction slide: if traction field is empty, note placeholder explicitly; do not invent numbers

### 6.3 Email Launch Sequence

**Template:** `templates/email-sequence.md`

**5-email structure:**

| Email | Timing | Subject line formula | Body structure | Goal |
|-------|--------|---------------------|---------------|------|
| E1 — Teaser | T-14 | ≤ 40 chars, curiosity gap, no product name | Problem story (no solution). End: "Stay tuned." | Open curiosity loop |
| E2 — Agitation | T-10 | "The [pain] problem is worse than you think" | Data point + customer story + blurred screenshot hint | Urgency around problem |
| E3 — Reveal | T-7 | "Here's what we've been building" | Name product + 1–2 sentence description + demo GIF descriptor + waitlist CTA | Intent |
| E4 — Proof | T-3 | "[Beta tester] said something that surprised us" | Testimonial quote + scarcity signal + reserve CTA | Reduce risk + scarcity |
| E5 — Launch | T-0 | "We're live" / "[Product] is open" | Live announcement + primary CTA + early-bird offer (48h expiry) + founder note | Convert |

**Per-email required fields:**
- Subject line (≤ 40 characters — enforced by validate-lengths.py)
- Preview text (≤ 90 characters)
- Opening line (hook — cannot start with "I" or the product name)
- Body (150–300 words)
- CTA (outcome language, 2–5 words)
- P.S. line (optional but recommended for E1, E3, E5)

**Hard constraints:**
- E1 and E2: product name must not appear
- E4: must include a quoted testimonial — even if placeholder, mark it `[TESTIMONIAL NEEDED]`
- E5: early-bird offer must have explicit expiry — "until Friday" not "limited time"

### 6.4 Product Hunt Listing

**Template:** `templates/product-hunt.md`

**4 components:**

```
TAGLINE
  [VARIABLE] — ≤ 60 characters. Formula: [Outcome verb] [what] [for whom] without [friction]
  Anti-pattern: Do not start with the product name. Do not use "AI-powered".

DESCRIPTION
  [VARIABLE] — ≤ 260 characters. 2 sentences: (1) what it does, (2) why now.
  Must include the unique_mechanism in plain language.

FIRST COMMENT (maker intro — posted at launch)
  [FIXED structure]
  Paragraph 1: Personal story — why this problem matters to you
  Paragraph 2: What you built and the unique mechanism
  Paragraph 3: What you're looking for (feedback, specific use case users)
  CTA: Link + early-bird offer
  Length: 150–250 words. Conversational, first-person, no marketing speak.

MAKER COMMENT (response template — for replying to upvoters)
  [VARIABLE] — 2–3 sentence thank-you + 1 specific question to continue conversation
  Anti-pattern: Do not write a generic "thanks for the support!" — ask something specific.
```

**Hard constraints:**
- Tagline: validated by validate-lengths.py (≤ 60 chars)
- Description: validated by validate-lengths.py (≤ 260 chars)
- First comment: must not read like marketing copy — flag any sentence with "revolutionary", "game-changer", "seamless", "robust"

### 6.5 Social Media Calendar

**Template:** `templates/social-calendar.md`

**30-day structure anchored to launch_date:**

| Phase | Days | Count | Theme |
|-------|------|-------|-------|
| Pre-launch | T-30 to T-21 | 4 posts | Foundation: problem awareness, founder story |
| Pre-launch | T-20 to T-8 | 6 posts | Awareness: teasers, reveals, countdown |
| Launch week | T-7 to T-1 | 5 posts | Heat: feature spotlight, testimonials, "3 days left" |
| Launch day | T-0 | 1 post | Announcement — platform-customized |
| Post-launch | T+1 to T+7 | 4 posts | Reception: results, FAQ, thank-you |
| Post-launch | T+8 to T+30 | 10 posts | Depth: case study, feature spotlights, social proof accumulation |

**Per-post required fields:**

```
Day: [relative day, e.g., T-21]
Date: [absolute date, if launch_date provided]
Platform: [Twitter/X | LinkedIn | Instagram | YouTube]
Post type: [problem | teaser | reveal | testimonial | feature | result | case-study]
Copy: [full post text]
Visual descriptor: [what the image/video should show — not a prompt, a descriptor]
CTA: [link | bio | comment prompt | none]
```

**Content mix rule (enforced in QA Layer 1):**
- 70% value (problem, education, story, result)
- 20% curated/engagement
- 10% promotional (direct product mention)
- If > 5 posts are promotional: Layer 1 structural fail

**Platform-specific copy rules:**

| Platform | Length | Format |
|----------|--------|--------|
| Twitter/X | ≤ 280 chars for single; 7-tweet thread for threads | Hook tweet must stand alone |
| LinkedIn | 150–300 words | No bullet points in first 3 lines (hidden by "see more") |
| Instagram | Caption 100–150 words | Visual descriptor required — copy is secondary |

### 6.6 Demo Script

**Template:** `templates/demo-script.md`

**Structure:**

```
[FIXED] HOOK (0:00–0:20)
  [VARIABLE] Opening question or bold claim — names the problem in 1 sentence
  [VARIABLE] Stakes — what it costs the ICP not to solve this
  Transition: "Let me show you how [Product] changes that."

[FIXED] CONTEXT SETUP (0:20–0:40)
  [VARIABLE] Describe the "before" state — what the user currently has to do
  Show the product interface: "Here's [Product]..."

[FIXED] CORE FEATURE (0:40–1:40)
  [VARIABLE] Step 1 — primary action a user takes
  [VARIABLE] Step 2 — the system's response / the magic
  [VARIABLE] Step 3 — the output / result
  Narration: outcome language, not feature language ("you now have X" not "the system processes Y")

[FIXED] WOW MOMENT (1:40–2:20)
  [VARIABLE] The single most impressive capability — something that makes the viewer say "I didn't know it could do that"
  Must be shown, not told. Describe exactly what appears on screen.
  Transition: "And this is why [unique_mechanism] matters."

[FIXED] SOCIAL PROOF BRIDGE (2:20–2:40)
  [VARIABLE] 1 quote from a real or representative user — specific outcome
  "[User] used [Product] to [specific result] in [timeframe]."

[FIXED] CTA (2:40–3:00)
  [VARIABLE] Restate solution_promise from spine
  [VARIABLE] Canonical CTA — outcome language
  [VARIABLE] URL or next step
```

**Hard constraints:**
- Target duration: 2:30–3:30 (validated by word count: 375–525 words at 150 wpm)
- Wow moment must demonstrate the unique_mechanism — not just a nice UI
- No slide transitions or marketing language in the script — it's a narration, not a pitch

## 7. Quality Gate

### Layer 1 — Structural (12 checks, deterministic)

All 12 must pass. Record all before writing the report.

| # | Check | Threshold |
|---|-------|-----------|
| L1-1 | Landing page hero complete | headline + subheadline + CTA + social proof hook all present |
| L1-2 | Landing page benefit sections | exactly 3 sections, each with 3 rows |
| L1-3 | Landing page FAQ | 3–5 Q&A pairs |
| L1-4 | Pitch deck slide count | exactly 12 slides |
| L1-5 | Pitch deck slide completeness | each slide has title + bullets + speaker note |
| L1-6 | Email count | exactly 5 emails |
| L1-7 | Email completeness | each email has subject + preview text + body + CTA |
| L1-8 | Product Hunt character limits | tagline ≤ 60, description ≤ 260 (use validate-lengths.py) |
| L1-9 | Social calendar count | exactly 30 posts, phase split 10/5/15 (pre/launch/post) ± 1 |
| L1-10 | Social calendar completeness | each post has platform + copy + visual descriptor |
| L1-11 | Social content mix | promotional posts ≤ 30% (≤ 9 of 30) |
| L1-12 | Demo script timing | 375–525 words (use validate-lengths.py) |

### Layer 2 — Narrative (8 dimensions, judgment)

Rating scale: **Strong (3) | Adequate (2) | Weak (1) | Failing (0)**

| Dim | Name | Key question |
|-----|------|-------------|
| N1 | Voice Consistency | Does the same person seem to have written all 6 deliverables? Read landing page headline then email E1 subject — do they sound like the same brand? |
| N2 | Problem-First Framing | Do all top-of-funnel assets (LP hero, E1 subject, first social posts, PH tagline) open on the problem, not the product? |
| N3 | Differentiation Clarity | Is the unique_mechanism stated once crisply and referenced in each deliverable? Or does it get buried or repeated verbatim? |
| N4 | Social Proof Placement | Does proof appear before every ask? (LP: proof before final CTA; E4: before reserve CTA; demo: before final CTA) |
| N5 | CTA Coherence | Do all CTAs point to the same action and use outcome language? Flag any "Sign up" or "Learn more" — they lose clicks. |
| N6 | Narrative Thread | Can you trace problem → insight → solution → proof → action through all 6 assets in sequence? State the thread explicitly in the report. |
| N7 | ICP Specificity | Does copy speak to one specific person or to everyone? Quote any sentence that addresses a generic audience ("businesses", "teams", "people"). |
| N8 | Generic Language | Flag: "revolutionary", "game-changing", "seamless", "robust", "powerful", "AI-powered", "best-in-class", feature-not-benefit statements. Count instances across all 6. |

**For every dimension rated Weak or Failing:** quote the specific offending line and write the specific fix. "This section could be stronger" is not permitted.

### Verdict logic

| Verdict | Condition |
|---------|-----------|
| **LAUNCH-READY** | Layer 1: 12/12 AND Layer 2: 0 Failing, ≤ 2 Weak |
| **REVISE-THEN-LAUNCH** | Layer 1: 12/12 AND Layer 2: 1 Failing OR 3+ Weak |
| **REWORK-REQUIRED** | Layer 1: any FAIL OR Layer 2: 2+ Failing |

## 8. Anti-Patterns

| Do NOT | Instead |
|--------|---------|
| Generate deliverables without stating the spine | Derive and show spine first; get confirmation before generating |
| Use generic copy formulas without ICP-specific language | Every headline must name the ICP's problem in their language |
| Invent traction numbers when traction field is empty | Mark as `[TRACTION NEEDED]` and flag in QA report |
| Skip Layer 2 because Layer 1 passed | Both layers always run — structural correctness ≠ quality |
| Produce vague N-layer feedback ("feels generic") | Quote specific line, name specific fix |
| Make 30 social posts that are variations of "check out our product" | Derive distinct content pillars — problem, story, feature, proof, behind-scenes |
| Use "revolutionary", "game-changing", "seamless" anywhere | Flag and rewrite with specific outcomes |
| Write the demo script as a pitch | It's a narration — outcome language, no superlatives, show don't tell |
| Treat the unique_mechanism as optional | If it's weak or generic, halt at SPINE phase and ask the user to sharpen it |
| Run generate before the user has confirmed the spine | The spine is the single point of failure — a wrong spine = 6 wrong deliverables |

## 11. Differentiation from Generic Launch Tools

Most AI-generated launch packages fail for two reasons:

1. **They generate deliverables independently** — the landing page uses different language than the email sequence, which uses different language than the pitch deck. There is no story, just content.

2. **They use copy templates as mad-libs** — slot in product name, job description, industry. The output is structurally correct and narratively empty.

This skill's answer to both: the **Narrative Spine** is derived before any template is touched. The spine is a forcing function — if the unique_mechanism is weak, the spine derivation fails visibly before 6 deliverables of weak copy are produced. The canonical language field ensures the same sentence architecture propagates from landing page to demo script.

The two-layer quality model — borrowed from seo-article-audit — separates structural compliance (does the package contain the right parts?) from narrative quality (is the story any good?). A package can pass Layer 1 and fail Layer 2. Both checks are always required.

## 12. Open Design Questions

These should be resolved before writing SKILL.md:

| Question | Options | Default |
|----------|---------|---------|
| Does the skill produce fully-written copy or structured outlines? | Full copy (longer, riskier) vs outlines with example lines | **Full copy** — the user's request implies production-ready output |
| Should the spine be a separate file or inline in SKILL.md? | Inline (simpler) vs `references/narrative-spine.md` (reusable) | **Separate reference** — other skills could reuse the spine derivation |
| Should the 30-day social calendar include platform-specific copy variants per post? | One copy per post vs per-platform variants | **One copy with platform adaptation notes** — full variants would be 90+ items |
| Should validate-lengths.py accept stdin or file path? | Stdin (composable) vs file path (simpler) | **File path** — easier for non-technical users |
| Should evals test the full 6-deliverable output or individual templates? | Full output (slower, more realistic) vs unit tests (faster) | **Full output for 2 cases** — the integration is the point |
